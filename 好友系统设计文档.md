# 好友系统设计文档

## 一、概述

好友系统建立在现有账号系统之上，让玩家能够跨对局维持社交关系，解决"靠口头报房间 ID 凑局"的体验问题。核心价值：

- **一键邀请**：在好友列表点击邀请，对方收到通知，无需手动报房间号
- **在线状态**：知道好友是否在线、是否在局中
- **持久关系**：与账号绑定，不随角色切换而消失

> 友谊基于**账号**（username），而非角色名。邀请时服务端使用当前角色名作为游戏内 playerId。

---

## 二、功能列表

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 搜索用户 | 按用户名搜索账号 | 高 |
| 发送好友申请 | 向对方发送申请，对方收到通知 | 高 |
| 接受/拒绝申请 | 接受后双向建立好友关系 | 高 |
| 好友列表 | 显示所有好友 + 在线/离线/局中状态 | 高 |
| 移除好友 | 单方删除，双向解除 | 高 |
| 邀请进房 | 在房间等待页邀请好友，对方客户端收到弹窗 | 高 |
| 撤回申请 | 在对方未回应前可撤销 | 中 |
| 好友申请列表 | 查看待处理的收到/发出申请 | 中 |

## 三、数据库设计

新增两张表（`apps/server/src/db/schema.sql`）：

```sql
-- 好友关系表（已通过审批的双向关系）
CREATE TABLE IF NOT EXISTS friendships (
  account_id_a TEXT NOT NULL,   -- 字典序小的一方
  account_id_b TEXT NOT NULL,   -- 字典序大的一方
  created_at   INTEGER NOT NULL,
  PRIMARY KEY (account_id_a, account_id_b)
);

-- 好友申请表（单向，待处理）
CREATE TABLE IF NOT EXISTS friend_requests (
  id             INTEGER PRIMARY KEY AUTOINCREMENT,
  from_account_id TEXT NOT NULL,
  to_account_id   TEXT NOT NULL,
  created_at      INTEGER NOT NULL,
  UNIQUE(from_account_id, to_account_id)
);
```

**查询好友列表**（单次查两侧）：
```sql
SELECT account_id_b AS friend_id FROM friendships WHERE account_id_a = ?
UNION
SELECT account_id_a AS friend_id FROM friendships WHERE account_id_b = ?
```

## 四、协议设计

### 4.1 新增类型

```typescript
// packages/protocol/src/index.ts

interface FriendInfo {
  accountId: string;
  username: string;
  online: boolean;          // 当前是否在线
  inRoom: boolean;          // 是否在游戏房间中
  pendingRequest?: "sent" | "received";  // 若是好友申请中的用户
}

interface RoomInviteInfo {
  fromAccountId: string;
  fromUsername: string;
  roomId: string;
}
```

### 4.2 客户端 → 服务器（新增事件）

```typescript
// 搜索用户（按用户名）
interface SearchUserEvent       { type: "searchUser";            token: string; username: string }
// 发送好友申请
interface SendFriendRequestEvent { type: "sendFriendRequest";    token: string; toAccountId: string }
// 接受好友申请
interface AcceptFriendRequestEvent { type: "acceptFriendRequest"; token: string; fromAccountId: string }
// 拒绝/撤回好友申请
interface DeclineFriendRequestEvent { type: "declineFriendRequest"; token: string; fromAccountId: string }
// 删除好友
interface RemoveFriendEvent      { type: "removeFriend";         token: string; accountId: string }
// 查看好友列表（含待处理申请）
interface ListFriendsEvent       { type: "listFriends";          token: string }
// 邀请好友进入当前房间
interface InviteFriendToRoomEvent { type: "inviteFriendToRoom";  token: string; toAccountId: string }
```

### 4.3 服务器 → 客户端（新增事件）

```typescript
// 好友列表（含在线状态）
interface FriendListEvent {
  type: "friendList";
  friends: FriendInfo[];
  pendingReceived: FriendInfo[];   // 收到的待回应申请
  pendingSent: FriendInfo[];       // 我发出的待回应申请
}

// 好友操作结果（通用）
interface FriendActionResultEvent {
  type: "friendActionResult";
  action: "request_sent" | "accepted" | "declined" | "removed" | "search_result";
  ok: boolean;
  error?: string;
  data?: FriendInfo[];             // 搜索结果或更新后的好友信息
}

// 推送：收到好友申请（实时通知）
interface FriendRequestNotifyEvent {
  type: "friendRequestNotify";
  from: { accountId: string; username: string };
}

// 推送：好友在线状态变化
interface FriendStatusEvent {
  type: "friendStatus";
  accountId: string;
  online: boolean;
  inRoom: boolean;
}

// 推送：收到房间邀请
interface RoomInviteEvent {
  type: "roomInvite";
  from: { accountId: string; username: string };
  roomId: string;
  expiresIn: number;   // 过期时间（ms），客户端倒计时后自动关闭弹窗
}
```

## 五、服务端架构

### 5.1 新增文件

```
apps/server/src/
  friend.ts           ← CRUD：添加/查询/删除好友关系、好友申请
  friend-notify.ts    ← 推送：好友上线/下线通知、房间邀请推送
```

### 5.2 friend.ts 关键函数

```typescript
// 获取所有好友的 accountId 列表
async function getFriends(accountId: string): Promise<string[]>

// 是否已是好友
async function areFriends(a: string, b: string): Promise<boolean>

// 发送好友申请（检查重复、已是好友等）
async function sendRequest(fromId: string, toId: string): Promise<{ ok: boolean; error?: string }>

// 接受申请：删除 friend_requests 记录，插入 friendships 记录
async function acceptRequest(fromId: string, toId: string): Promise<void>

// 删除好友（删除 friendships 双向记录）
async function removeFriend(a: string, b: string): Promise<void>
```

### 5.3 在线状态推送时机

| 触发事件 | 推送内容 | 接收者 |
|---------|---------|--------|
| 玩家完成认证（login/token恢复） | `friendStatus { online: true }` | 该玩家的所有在线好友 |
| 玩家 WebSocket 断线（grace期后确认） | `friendStatus { online: false }` | 该玩家的所有在线好友 |
| 玩家加入/离开房间 | `friendStatus { inRoom: true/false }` | 该玩家的所有在线好友 |

**实现要点**：需要一个 `accountId → PlayerConn` 的映射（现有 `playersById` 是按 playerId 索引的，需要额外维护 `connsByAccountId: Map<string, PlayerConn>`）。

### 5.4 房间邀请流程

```
玩家A（在房间中）→ inviteFriendToRoom { toAccountId: B }
  → 服务端检查：B 是否是 A 的好友
  → 服务端检查：B 是否在线且不在其他游戏中
  → 找到 B 的 PlayerConn
  → 向 B 发送 roomInvite 事件
  → B 客户端弹窗显示邀请（30s 后过期）
  → B 点击"接受" → 客户端发送 joinRoom { roomId }
```

## 六、客户端架构

### 6.1 新增组件

```
apps/client/src/
  components/
    FriendPanel.tsx        ← 好友列表面板（折叠式浮窗，类似 ChatWindow）
    FriendSearch.tsx       ← 用户名搜索 + 发送申请

```

### 6.2 Store 新增字段

```typescript
// gameStore.ts 新增
friends: FriendInfo[];
pendingFriendRequests: { received: FriendInfo[]; sent: FriendInfo[] };
pendingRoomInvite: RoomInviteInfo | null;  // 当前收到的房间邀请

// 事件处理
"friendList"          → 更新 friends + pendingFriendRequests
"friendRequestNotify" → 显示 Toast 通知 + 更新 pending
"friendStatus"        → 更新 friends 中对应项的在线状态
"roomInvite"          → 设置 pendingRoomInvite，显示弹窗
"friendActionResult"  → 处理各类操作结果
```

### 6.3 FriendPanel 位置

与 ChatWindow 类似，固定右下角浮窗，但放在 ChatWindow **上方**（`bottom: 480px` or 按展开高度动态偏移）。

在**所有视图**（登录后）均可访问，不仅限于游戏中。

## 七、实施顺序

```
阶段 1：基础关系（2-3天）
  ├── DB schema + friend.ts CRUD
  ├── 协议：listFriends / sendRequest / acceptRequest / declineFriendRequest / removeFriend
  ├── server/index.ts 路由
  └── FriendPanel.tsx（静态显示好友列表，无实时状态）

阶段 2：实时状态（1-2天）
  ├── connsByAccountId 映射
  ├── 登录/断线时推送 friendStatus
  └── FriendPanel 实时更新在线状态

阶段 3：房间邀请（1天）
  ├── inviteFriendToRoom 事件
  ├── RoomInviteToast.tsx
  └── 接受邀请 → 自动 joinRoom

阶段 4：打磨
  ├── 搜索用户（searchUser）
  ├── 好友申请通知（friendRequestNotify 推送）
  └── 申请未读徽标
```

## 八、已知限制与边界

| 问题 | 说明 |
|------|------|
| 单账号多标签页 | 同一账号多个浏览器标签均连接时，好友状态以"最后一个连接"为准；多连接管理较复杂，Phase 1 不处理 |
| 好友上限 | 建议设上限（如 200），防止数据库查询过慢 |
| 跨设备通知 | 当前无移动端推送，仅 WebSocket 实时通知 |
| 房间邀请过期 | 若 B 在邀请过期前关闭页面，服务端不需要处理（客户端过期后不发送 joinRoom 即可） |
| 好友申请轰炸 | 需要限速（如同一账号对同一目标 1 小时内只能发 1 次申请）|
