# 聊天系统设计文档

## 一、概述

本文档规划 ProGyakuten 的队内沟通系统，分为两个子系统：

- **文字聊天**：队友之间发送文字消息，实时显示在游戏界面
- **语音通话**：4/6 人局中，同队玩家开麦实时语音沟通

两个子系统相互独立，可分阶段实现。文字聊天优先级更高，实现成本低；语音通话涉及 WebRTC，复杂度显著更高。

> 根据游戏规则：同队玩家手牌互见，可讨论战术；对方队员仅知道手牌数量，不可听见队内讨论内容。因此，文字聊天和语音通话均为**仅队友可见/可听**。

## 二、文字聊天

### 2.1 功能描述

- 玩家在游戏中可向队友发送文字消息
- 消息仅本队玩家可见
- 消息显示发送者名字 + 内容 + 时间戳
- 对局结束后聊天记录不保留
- 2人局（1v1）禁用或提示"单人队伍无需聊天"

### 2.2 协议设计

新增两个事件（加入 `packages/protocol/src/index.ts`）：

```typescript
// 客户端 → 服务器
interface TeamChatEvent {
  type: "teamChat";
  message: string;           // 消息内容（服务端截断至 200 字）
}

// 服务器 → 客户端（广播给同队所有人，含发送者自己）
interface TeamChatMessageEvent {
  type: "teamChatMessage";
  fromPlayerId: string;
  message: string;
  timestamp: number;         // 服务器时间戳（ms）
}
```

`ClientEvent` 联合类型加入 `TeamChatEvent`；`ServerEvent` 联合类型加入 `TeamChatMessageEvent`。

### 2.3 服务端实现

在 `apps/server/src/handler.ts` 中处理 `teamChat` 事件：

```
收到 teamChat
  → 找到发送者所在队伍（teamA / teamB）
  → 取同队所有玩家连接
  → 向每人发送 teamChatMessage（含发送者本人）
```

注意事项：
- 仅 `in_game` 状态的房间处理聊天
- 消息长度在服务端截断（200 字符）
- 不持久化到 SQLite（聊天是临时数据）

### 2.4 客户端实现

**Store 新增字段（`gameStore.ts`）：**

```typescript
chatMessages: { fromPlayerId: string; message: string; timestamp: number }[];
addChatMessage: (msg: ...) => void;
```

`applyEvent` 新增 `"teamChatMessage"` case，调用 `addChatMessage`。

**UI（`GameBoard.tsx`）：**

在日志面板旁或合并为"日志/聊天"标签页，新增聊天面板：

```
┌──────────────────────────────┐
│ 日志 │ 队伍聊天               │  ← 标签切换
├──────────────────────────────┤
│ 队友A: 打红色5                │
│ 你: 好的，我来抢              │
│ 队友A: ✓                     │
├──────────────────────────────┤
│ [输入消息...      ] [发送]    │
└──────────────────────────────┘
```

快捷键：Enter 发送，消息框聚焦时不触发游戏快捷键。

### 2.5 新增文件

| 文件 | 说明 |
|------|------|
| `apps/client/src/components/ChatPanel.tsx` | 聊天面板组件（消息列表 + 输入框） |

## 三、语音通话

### 3.1 功能描述

- 4/6 人局中，同队玩家可开麦实时通话
- 开麦/关麦按钮（Push-to-Talk 或 Toggle 两种模式，可配置）
- 游戏界面显示各队友的发言状态（声波动画图标）
- 队友静音状态可见（红色麦克风图标）
- 仅本队玩家的音频互通，对方队伍完全隔离

### 3.2 技术选型

#### 推荐方案：WebRTC + 现有 WebSocket 作信令通道

```
玩家A ←──── WebRTC 音频流（P2P 直连）────→ 队友B
 ↓ SDP/ICE 信令                              ↑
 └──────────── 现有 WS 服务器中转 ───────────┘
```

**为什么选 WebRTC？**

| 特点 | 说明 |
|------|------|
| 零额外服务器成本 | 利用已有 WS 服务器做信令；音频 P2P 传输 |
| 浏览器原生支持 | 无需额外 SDK，使用 `RTCPeerConnection` API |
| 延迟低 | P2P 直连，延迟 50-100ms |
| 规模匹配 | 队伍最多 3 人，全网状连接数最多 3 条，开销极小 |

**已知局限：**
- 部分对称型 NAT 网络无法 P2P，需要 TURN 服务器兜底（初期可先不配置，仅局域网/良好 NAT 有效）
- 需要浏览器麦克风权限
- 移动端浏览器兼容性较桌面端差

#### 备选方案对比

| 方案 | 成本 | 复杂度 | 适合时机 |
|------|------|--------|---------|
| WebRTC P2P（推荐） | 零 | 中 | 当前 |
| WebRTC + SFU（mediasoup/LiveKit） | 服务器资源 | 高 | 用户量上来后 |
| Agora/Daily.co 等第三方 | 免费额度+付费 | 低 | 想快速落地且接受依赖 |
| WebSocket 音频中继 | 带宽高 | 低 | 不推荐（占用过多 WS 带宽） |

### 3.3 WebRTC 拓扑

队伍人数最多 3 人（6人局），全网状连接：

```
2人队: A ──── B         (1条连接)
3人队: A ──── B         (3条连接)
           ╲ ╱
            C
```

对于 ≤3 人的全网状，浏览器负载可忽略不计。

### 3.4 STUN / TURN 配置

```
ICE 服务器配置（客户端）：

iceServers: [
  { urls: "stun:stun.l.google.com:19302" },      // 免费 STUN
  { urls: "stun:stun1.l.google.com:19302" },
  // TURN 服务器（初期留空，后期按需加入）：
  // { urls: "turn:...", username: "...", credential: "..." }
]
```

**TURN 服务器选项（需要时再接入）：**
- **Twilio TURN**：有免费额度，按用量计费
- **自建 coturn**：在服务器或 VPS 上运行，完全免费
- **Metered TURN**：有免费套餐（50GB/月）

初期可先上线文字聊天 + STUN-only 语音（局域网和良好 NAT 环境下完全可用），待有用户反馈再加 TURN。

### 3.5 信令协议设计

在 `packages/protocol/src/index.ts` 新增信令事件：

```typescript
// ── 客户端 → 服务器 ──────────────────────────────────────────

interface VoiceJoinEvent {
  type: "voiceJoin";           // 申请加入队内语音房间
}

interface VoiceLeaveEvent {
  type: "voiceLeave";          // 离开语音房间（关麦或离线）
}

interface VoiceOfferEvent {
  type: "voiceOffer";
  toPlayerId: string;          // 向特定队友发送 SDP offer
  sdp: string;
}

interface VoiceAnswerEvent {
  type: "voiceAnswer";
  toPlayerId: string;
  sdp: string;
}

interface VoiceIceCandidateEvent {
  type: "voiceIceCandidate";
  toPlayerId: string;
  candidate: string;           // JSON.stringify(RTCIceCandidate)
}

interface VoiceMuteEvent {
  type: "voiceMute";
  muted: boolean;              // 广播自己的静音状态给队友
}

// ── 服务器 → 客户端 ──────────────────────────────────────────

interface VoicePeerJoinedEvent {
  type: "voicePeerJoined";
  playerId: string;            // 队友加入了语音房间
}

interface VoicePeerLeftEvent {
  type: "voicePeerLeft";
  playerId: string;
}

// 以下三个是信令中转，fromPlayerId 由服务端填充
interface VoiceOfferRelayEvent {
  type: "voiceOfferRelay";
  fromPlayerId: string;
  sdp: string;
}

interface VoiceAnswerRelayEvent {
  type: "voiceAnswerRelay";
  fromPlayerId: string;
  sdp: string;
}

interface VoiceIceCandidateRelayEvent {
  type: "voiceIceCandidateRelay";
  fromPlayerId: string;
  candidate: string;
}

interface VoiceMuteStatusEvent {
  type: "voiceMuteStatus";
  playerId: string;
  muted: boolean;              // 广播队友静音状态变化
}
```

### 3.6 服务端信令中转（新文件）

新建 `apps/server/src/voice.ts`：

**RoomState 新增字段：**
```typescript
voicePlayers?: Set<string>;   // 当前在语音房间里的玩家
voiceMuteState?: Record<string, boolean>;  // 每人的静音状态
```

**信令处理逻辑（伪代码）：**

```
收到 voiceJoin：
  room.voicePlayers.add(playerId)
  向同队其他 voicePlayers 广播 voicePeerJoined
  让刚加入者知道现有 voicePlayers 列表（发一个 voiceRoomState）

收到 voiceOffer / voiceAnswer / voiceIceCandidate：
  找到 toPlayerId 的连接
  校验 toPlayerId 与发送者是同队
  转发给 toPlayerId（附上 fromPlayerId）

收到 voiceMute：
  更新 room.voiceMuteState[playerId]
  向同队 voicePlayers 广播 voiceMuteStatus

收到 voiceLeave / 断线：
  room.voicePlayers.delete(playerId)
  向同队广播 voicePeerLeft
```

**关键约束：**
- 信令转发只在同队玩家之间（防止信息泄漏）
- 每条信令转发前验证发送者和目标是否同队

### 3.7 客户端 WebRTC 管理（新文件）

新建 `apps/client/src/utils/voice.ts`，导出 `VoiceManager` 类：

```typescript
class VoiceManager {
  private peers = new Map<string, RTCPeerConnection>();
  private localStream: MediaStream | null = null;
  private onMuteChange: (muted: boolean) => void;

  // 获取麦克风权限并加入
  async join(): Promise<void>

  // 离开并关闭所有连接
  leave(): void

  // 切换静音状态
  toggleMute(): boolean

  // 当有新队友加入时，作为 offerror 建立连接
  async handlePeerJoined(peerId: string): Promise<void>

  // 处理来自服务器的信令消息
  async handleOffer(fromId: string, sdp: string): Promise<void>
  async handleAnswer(fromId: string, sdp: string): Promise<void>
  async handleIceCandidate(fromId: string, candidateStr: string): Promise<void>

  // 关闭与特定队友的连接
  closePeer(peerId: string): void
}
```

**RTCPeerConnection 创建模板：**

```typescript
const pc = new RTCPeerConnection({
  iceServers: [
    { urls: "stun:stun.l.google.com:19302" },
    { urls: "stun:stun1.l.google.com:19302" }
  ]
});

// 本地音频轨道（开启降噪、回声消除）
const stream = await navigator.mediaDevices.getUserMedia({
  audio: {
    echoCancellation: true,
    noiseSuppression: true,
    autoGainControl: true
  },
  video: false
});

stream.getAudioTracks().forEach(track => pc.addTrack(track, stream));

// 远端音频输出
pc.ontrack = (e) => {
  const audio = new Audio();
  audio.srcObject = e.streams[0];
  audio.autoplay = true;
};
```

## 四、实施顺序（推荐）

```
阶段 1：文字聊天（2-3天）
  ├── protocol: +TeamChatEvent / +TeamChatMessageEvent
  ├── server: handler.ts 中路由 teamChat
  ├── client: gameStore + ChatPanel.tsx
  └── styles: 聊天面板样式

阶段 2：语音协议层（1天）
  ├── protocol: +全部 Voice* 事件
  └── server: types.ts 加 voicePlayers / voiceMuteState 字段

阶段 3：语音服务端（2天）
  ├── server: voice.ts 信令中转处理器
  └── server: index.ts 加路由

阶段 4：语音客户端（3-4天）
  ├── client: utils/voice.ts（VoiceManager 类）
  ├── client: gameStore 加 voice 状态
  ├── client: VoiceControls.tsx
  └── client: GameBoard.tsx 集成

阶段 5：测试与打磨（1-2天）
  ├── LAN 测试（最简单情况）
  ├── 跨网络测试（STUN 是否够用）
  └── 断线重连后语音重建
```

## 五、已知限制与风险

| # | 问题 | 影响 | 缓解方案 |
|---|------|------|---------|
| V-1 | 对称型 NAT 下 P2P 无法建立 | 部分用户语音无法工作 | 后期接入 TURN 服务器 |
| V-2 | 浏览器麦克风权限 | 用户需手动允许 | UI 明确引导权限请求 |
| V-3 | Render 免费层冷启动 15 分钟后休眠 | 语音信令依赖 WS，冷启动后 WebRTC 连接失效 | 语音重连时触发重新协商；WS 重连后自动重新 `voiceJoin` |
| V-4 | 移动端兼容性 | iOS Safari 对 WebRTC 有部分限制 | 明确"桌面端体验最佳"；移动端降级为只用文字聊天 |
| V-5 | 回声问题 | 使用外放扬声器时，对方会听到回声 | `echoCancellation: true` 可缓解；建议用户使用耳机 |
| V-6 | 带宽占用 | 音频编码约 30-50 kbps/路，3条连接约 150 kbps | Opus 编码默认开启，已足够低；告知用户低速网络下体验可能下降 |
| V-7 | 信令服务器单点故障 | WS 服务器挂掉，语音信令也断 | WebRTC 一旦建立，P2P 流不经过服务器，信令断开后音频短期内仍可通 |

## 六、不做的事（范围边界）

- ❌ 全局语音（两队互通）——违反游戏规则，永久不做
- ❌ 游戏录音/回放——隐私和存储成本均不合适
- ❌ 语音转文字——成本过高，不在计划内
- ❌ 观战者语音——观战功能本身尚未实现，优先级更低
- ❌ 服务端 SFU（mediasoup 等）——当前用户量下过度设计，等有需要再升级
