# 观战系统设计文档

## 一、概述

观战系统允许未参与对局的玩家以**只读视角**旁观正在进行的游戏。核心约束：

- **公平原则**：观战者看不到任何玩家的具体手牌，只能看到手牌数量
- **无干预原则**：观战者无法发送任何游戏动作
- **实时同步**：观战者接收与玩家相同的公开状态更新
- **随时离开**：观战者可以任意加入或离开，不影响对局进行

观战系统与**好友系统**结合后，可以实现"观看好友的对局"，是提升社交黏性的重要功能。

## 二、功能设计

### 2.1 观战入口

| 入口 | 说明 |
|------|------|
| 大厅列表 | `in_game` 状态的房间显示"观战"按钮（替代灰色"进入"）|
| 好友列表 | 好友处于 `inRoom: true` 状态时，显示"观战"按钮 |

### 2.2 观战者视角

| 信息 | 是否可见 | 备注 |
|------|---------|------|
| 弃牌堆顶牌 | ✅ | |
| 摸牌堆数量 | ✅ | |
| 各玩家手牌**数量** | ✅ | |
| 各玩家手牌**内容** | ❌ | 保护竞技公平 |
| 当前出牌方 | ✅ | |
| 游戏阶段（主回合/抢牌/摸牌后） | ✅ | |
| 罚摸堆计数 | ✅ | |
| 对局日志/公告 | ✅ | 如"XXX 摸了 4 张牌" |
| 角色信息 | ✅ | 玩家选的角色名和技能 |
| 队伍分组 | ✅ | |

### 2.3 对局结束后

- 观战者看到结算画面
- 允许选择继续观战下一局（如果房间再开局），或主动离开
- **Phase 2 扩展**：结算后可展示所有玩家手牌（复盘视角）

### 2.4 观战者聊天

观战者可以使用**全员频道**与玩家（和其他观战者）交流，也有**观战专属频道**与其他观战者互动。

聊天系统的 `scope` 字段新增 `"spectator"`：
- `scope: "room"` → 全体可见（玩家 + 观战者）
- `scope: "spectator"` → 仅观战者可见

## 四、服务端架构

### 4.1 RoomState 新增字段

```typescript
export interface RoomState {
  // ... 现有字段
  spectators: string[];      // 观战者的 playerId 列表（不在 players 中）
}
```

### 4.2 核心逻辑：观战者广播

**原则**：游戏状态更新时，同时向观战者广播，但发送的 `statePatch` 中 `hand: []`。

在 `broadcast.ts` 新增：

```typescript
// 向所有观战者广播当前游戏状态（手牌字段为空）
export function broadcastSpectatorState(room: RoomState, message?: string): void {
  for (const spectatorId of room.spectators) {
    const conn = playersById.get(spectatorId);
    if (!conn || conn.roomId !== room.roomId) continue;
    send(conn.ws, buildSpectatorStateEvent(room, message));
  }
}

function buildSpectatorStateEvent(room: RoomState, message?: string): ServerEvent {
  return {
    type: "statePatch",
    state: toPublicState(room.game!),
    phase: room.phase!,
    hand: [],                  // ← 观战者看不到手牌
    teammateHands: {},         // ← 观战者看不到队友手牌
    message,
    allowedActions: [],        // ← 观战者无可用动作
    lastSeq: undefined
  };
}
```

**修改 `broadcastGameState`**，在内部同时广播给观战者：

```typescript
export function broadcastGameState(room, message?, presentationHint?) {
  // 现有逻辑：向玩家广播
  for (const playerId of room.players) { ... }

  // 新增：向观战者广播（无手牌）
  broadcastSpectatorState(room, message);
}
```

### 4.3 加入观战流程

```
客户端发送 joinRoomAsSpectator { roomId }
  → 校验：房间存在 / 不是已参与玩家 / 观战人数未超上限（如 20）
  → 设置 conn.roomId = roomId
  → room.spectators.push(conn.playerId)
  → 向该观战者发送 spectatorGameSnapshot（当前完整状态）
  → 向所有玩家 + 观战者广播 spectatorJoined
```

### 4.4 离开观战流程

```
主动离开（leaveSpectator）或 WebSocket 断线：
  → room.spectators = room.spectators.filter(id => id !== playerId)
  → 清理 conn.roomId
  → 向所有玩家 + 观战者广播 spectatorLeft
```

注意：观战者断线不影响对局，不需要重连机制。

### 4.5 新增文件

```
apps/server/src/
  spectator.ts      ← joinRoomAsSpectator / leaveSpectator 处理逻辑
```

修改文件：`broadcast.ts` / `connection.ts`（断线时检查 spectators）/ `index.ts`（路由新事件）

### 4.6 观战聊天的 scope 处理

```typescript
// server/index.ts 中的 chat 处理器

if (event.scope === "spectator") {
  // 只向观战者广播
  const recipients = room.spectators;
  ...
} else if (event.scope === "room") {
  // 向所有玩家 + 观战者广播（现有逻辑，需扩展 recipients）
  const recipients = [...room.players, ...room.spectators];
  ...
}
```

## 六、与现有系统的集成

### 6.1 与演出系统

观战者也应接收 `presentationHint`（演出触发），体验打牌音效和技能动画。

`broadcastSpectatorState` 中也传递 `presentationHint`：
```typescript
// 观战者也收到演出触发
send(conn.ws, { ...buildSpectatorStateEvent(room, message), presentationHint });
```

### 6.2 与角色系统

`spectatorGameSnapshot` 中包含 `characterAssignments`（或在 `GamePublicState` 中扩展）：
```typescript
// GamePublicState 可选扩展（Phase 2）
characterAssignments?: Record<string, CharacterPublicInfo>;
```

目前 `characterAssignments` 存在 `GameStateInternal` 中（只有 characterId，不含公开信息）。Phase 1 可先通过 `gameCharacterReveal` 事件（已有）传给观战者。

### 6.3 与好友系统

```
好友列表中好友状态 = "局中" → 显示"观战"按钮
点击"观战" → 发送 joinRoomAsSpectator { roomId: friend.roomId }
（需要 FriendInfo 中包含 roomId，当好友 inRoom = true 时填充）
```

## 八、已知限制

| 问题 | 说明 |
|------|------|
| 观战人数上限 | 建议限制 20 人，防止服务端广播压力过大 |
| 观战者无重连 | 断线后需重新加入，不保存观战上下文（代价低，合理）|
| 游戏中途加入 | 观战者加入时只能看到当前状态，看不到历史操作（需录像系统才能回放）|
| 信息安全 | 观战者 `hand = []`，但需确保服务端不会通过其他事件泄漏手牌数据 |
| 和 AI 玩家共存 | AI 玩家 `playerId` 以 `"AI_"` 开头，观战者列表需过滤 AI |
| 观战者对局体验 | 无法参与抢牌、UNO 检举等互动，观战略显被动——后续可考虑"观战弹幕/反应" |

## 十、已知 Bug 与待修复事项

| # | 问题 | 优先级 | 修复思路 |
|---|------|--------|---------|
| S-1 | **新一局开始时观战者不收到 `gameCharacterReveal`**：选角和角色揭示事件只发给 `room.players`，观战者的 `characterAssignments` 不会随新一局更新，角色面板显示仍是上一局数据（或空） | 中 | 在 `completeDraft` 和 `launchGame` 中，除向玩家发送外，也向 `room.spectators` 发送 `gameCharacterReveal` 和带 `characterAssignments` 的初始状态 |
| S-2 | **`leaveRoom` 事件对观战者无效**：观战者发送 `leaveRoom` 时，服务端查不到其在 `room.players` 中，什么都不做，客户端卡住 | 高 | 在 `index.ts` 的 `leaveRoom` 处理前，检查 `room.spectators.includes(conn.playerId)`，若是观战者则调用 `handleLeaveSpectator` |
| S-3 | **观战者使用聊天时，ChatWindow 的"队伍"tab 不应显示**：`ChatWindow.tsx` 的"队伍"tab 对观战者没有意义（服务端会拒绝其 team scope 消息），但 tab 仍显示可点击 | 低 | 在 `ChatWindow` 中接收 `isSpectating` prop，观战者时隐藏队伍 tab |
| S-4 | **观战者在 `game_over` 后的状态不确定**：游戏结束后，玩家可以"再来一局"，房间重新进入 `lobby` 然后 `in_game`。观战者的 `isSpectating=true` 状态保持，但 `gameState` 逐渐变为新游戏的状态。整体可用，但首屏可能短暂显示旧数据 | 低 | 在 `gameOver` handler 中，若 `isSpectating=true`，清除 `gameState` 并显示"等待下一局"提示，待收到新 `statePatch` 再恢复 |
| S-5 | **`broadcastRoomSnapshot` 不向观战者发送**：`room.ts` 中 `broadcastRoomSnapshot` 只遍历 `room.players`，观战者收不到房间人员变动通知 | 低 | 在 `broadcastRoomSnapshot` 中也遍历 `room.spectators`，向其发送 roomSnapshot |
