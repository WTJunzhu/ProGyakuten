# 角色系统设计文档

## 一、概述

参考三国杀机制：每局开始前，每位玩家从随机抽到的 3 张角色卡中选择 1 张，角色各有独特技能，影响当局游戏规则。

设计目标：**加入新角色只需新增一个角色定义文件 + 一行注册**，不改动任何主流程代码。

---

## 二、架构决策

### 技能 = 绑定玩家的规则钩子

项目已有 `GameRuleHookSet` 钩子系统（`canPlayCard` / `afterCardPlayed` 等）。  
每个技能通过 `createHooks(playerId)` 工厂函数，返回一组只对该玩家生效的钩子，在游戏开始时注入 `state.rules.hooks`。

被动技能 → 纯钩子（无需玩家操作）  
主动技能 → 钩子 + 新增 `UseSkillEvent` 动作（玩家点按钮触发）

## 四、核心类型定义

```typescript
// packages/core/src/characters/types.ts

/** 技能的客户端展示信息（不含逻辑） */
export interface SkillPublicInfo {
  id: string;
  name: string;
  description: string;
  isActive: boolean;
  maxUsesPerGame?: number;
  maxUsesPerTurn?: number;
}

/** 角色的客户端展示信息 */
export interface CharacterPublicInfo {
  id: string;
  name: string;
  description: string;
  skills: SkillPublicInfo[];
}

/** 技能完整定义（含逻辑，仅在 core/server 使用） */
export interface SkillDefinition extends SkillPublicInfo {
  // 返回绑定到特定玩家的规则钩子（被动技能必填）
  createHooks?: (playerId: string) => Partial<GameRuleHookSet>;
  // 主动技能：当前状态下能否激活
  canActivate?: (state: GameStateInternal, playerId: string) => boolean;
  // 主动技能：激活效果
  onActivate?: (state: GameStateInternal, playerId: string, payload?: unknown) => ActionResult;
}

/** 角色完整定义 */
export interface CharacterDefinition extends CharacterPublicInfo {
  skills: SkillDefinition[];
}
```

## 六、GameRuleHookSet 需要新增的钩子

现有钩子覆盖出牌/抢牌/UNO，角色技能需要补充：

| 新钩子 | 触发位置 | 用途举例 |
|--------|---------|---------|
| `afterTurnStart` | `startMainTurn()` 开头 | 诸葛亮：回合开始额外摸牌 |
| `afterCardDrawn` | `drawOne()` 末尾 | 曹操：摸到牌后可即时打出 |
| `beforePenaltyDraw` | `applyPassTurn()` 罚摸前 | 某角色：拦截/重定向罚摸 |
| `onSkipConstraintSet` | `applySkipEffect()` 末尾 | 某角色：免疫 Skip 约束 |

## 八、游戏生命周期变化

```
lobby
  ↓  房主点「开始游戏」
character_selection          ← 新增阶段（RoomState.status 新增此值）
  ↓  全员选完 or 30秒超时
in_game                      ← 正式开始（同现在）
```

**character_selection 阶段服务端流程：**

1. 从全部角色中随机抽取，确保每人的 3 张候选**互不重复**（跨玩家也不重复）
2. 向每位玩家**单独私发** `characterDraft`（其他人看不到你的选项）
3. 启动 30 秒倒计时（`setTimeout`，逻辑同现有 phase 超时机制）
4. 收到 `selectGameCharacter` → 记录选择，超时未选 → 随机分配一张
5. 全员完成 → 调用 `applyCharacterSkills(game, assignments)` 把技能钩子注入 `state.rules.hooks`
6. 广播 `gameCharacterReveal`（公开所有人的角色）
7. 正常走 `startMainTurn()` 开始游戏

## 九、主动技能的触发流程

```
客户端点「使用技能」按钮
  → 发送 UseSkillEvent { skillId, payload }
  → 服务器 handler.ts 路由到 character-draft.ts 的 handleUseSkill()
  → 查 registry 找到角色定义 → 调 skill.canActivate() 校验
  → 通过 → 调 skill.onActivate() 执行效果，扣除使用次数
  → 广播状态更新
```

客户端的 `allowedActions` 新增 `"use_skill"`，配合当前玩家可用的技能列表，有可激活技能时才显示按钮。

## 十一、角色文件格式规范（标准模板）

> 每个角色独占一个文件，放在 `packages/core/src/characters/definitions/` 下。  
> 加入新角色只需：① 新建文件 ② 在 `registry.ts` 末尾加一行 import + registerCharacter。

```typescript
// packages/core/src/characters/definitions/角色拼音或英文名.ts
import type { CharacterDefinition } from "../types";
import type { GameStateInternal, ActionResult } from "../../types";
// 若技能需要操作游戏内部状态，从 engine/state 按需引入：
// import { drawOne, ensureDrawPile } from "../../engine/state";

export const 角色变量名: CharacterDefinition = {
  id: "唯一英文id",           // 全局唯一，小写+下划线，如 "naruhodou"
  name: "角色显示名",          // 显示在 UI 上，如 "成步堂龙之介"
  description: "角色背景/简介", // 一两句话，显示在选角卡片上

  skills: [
    // ──── 被动技（每局自动监听，无需玩家点击） ────
    {
      id: "技能唯一id",         // 同一角色内唯一，如 "skill_passive_a"
      name: "技能名称",
      description: "技能说明（显示在悬停 tooltip 中）",
      isActive: false,          // false = 被动 / 点一下就触发（无额外选择步骤）
      // inputType 留空或 "none"，被动技无需玩家额外输入
      // maxUsesPerGame / maxUsesPerTurn 不填则无限次

      // 被动钩子：绑定到特定玩家，由游戏引擎自动回调
      createHooks: (playerId) => ({
        afterCardPlayed: ({ state, player, card }) => {
          if (player.playerId !== playerId) return;
          // 技能效果……
        },
        // 可同时注册多个钩子
      }),
    },

    // ──── 主动技（玩家点击立绘触发，可能需要选目标/选牌/选颜色） ────
    {
      id: "技能唯一id",
      name: "技能名称",
      description: "技能说明",
      isActive: true,           // true = 主动，需要玩家点击立绘触发
      inputType: "none",        // 见下方 inputType 说明
      maxUsesPerGame: 1,        // 每局限 1 次（不填 = 无限）

      // 主动技激活条件（服务端校验 + 客户端高亮立绘）
      canActivate: (state: GameStateInternal, playerId: string): boolean => {
        // 返回 true 时立绘高亮，玩家可点击触发
        return true; // 替换为实际条件
      },

      // 主动技效果（服务端执行）
      onActivate: (state: GameStateInternal, playerId: string, payload?: unknown): ActionResult => {
        // payload 由客户端根据 inputType 填入（目标id / 牌id / 颜色 等）
        return { ok: true, announcements: ["技能生效"] };
      },
    },
  ],
};
```

### inputType 取值说明

| 值 | 含义 | payload 格式 | 客户端 UX |
|---|---|---|---|
| `"none"` 或不填 | 无需额外输入，点立绘直接触发 | `{}` | 单次点击立绘 |
| `"target"` | 需要选择目标玩家 | `{ targetPlayerId: string }` | 点自己立绘 → 点目标立绘 |
| `"card"` | 需要从手牌中选一张牌 | `{ cardId: string }` | 点自己立绘 → 点手牌 |
| `"card_and_color"` | 选一张牌并指定颜色 | `{ cardId: string; newColor: CardColor }` | 点立绘 → 点手牌 → 颜色弹窗 |

### registry.ts 维护方式

```typescript
// packages/core/src/characters/registry.ts

// ── 内置角色 ── 每加一个角色在此处追加两行 ──────────────────

import { mikotoba } from "./definitions/mikotoba";
```

## 十三、已知复杂点（设计预留）

1. **曹操「奸雄」**：罚摸后的"即时出牌"需要在服务端新开一个小窗口（类似现有 `post_draw_window`），是所有角色里逻辑最复杂的，建议最后实现
2. **貂蝉「离间」**：`beforePenaltyDraw` 钩子需要携带"目标玩家可重定向"语义，协议上需要额外 `payload`
3. **主动技能 UI**：需要在 `GameBoard.tsx` 的操作区新增「使用技能」按钮区域（不涉及手牌区布局）
4. **断线重连**：`skillState` 需要跟随 `GameStateInternal` 一起序列化进 SQLite 快照，现有持久化层透明支持（已是 JSON 存储）

## 十四、已知健壮性问题（待修复）

> 以下问题已通过代码评审识别，按层分类，标注修复优先级（高/中/低）。

### 核心逻辑层

| # | 问题 | 优先级 | 修复思路 |
|---|------|--------|---------|
| C-1 | **成步堂判定触发时机不受限制**：`canActivate` 只检查"敌方有 1 张牌"，不检查当前阶段，在 `snatch_window` / `post_draw_window` 中也可触发，可能打断对局节奏 | 中 | 在 `canActivate` 中加 `state.turnPhase === "turn_main"` 检查，或明确允许"随时触发"并记入规则文档 |
| C-2 | **成步堂判定调用 `drawOne` 触发 `afterCardDrawn` 钩子**：判定效果对目标调用 `drawOne`，会触发所有注册了 `afterCardDrawn` 的角色被动（包括其他角色），形成意外连锁 | 高 | 新增 `drawOneQuiet(state, player)` 内部函数，不触发钩子；判定效果和惩罚性摸牌使用该函数 |
| C-3 | **御琴羽技能后 `penaltySource` 未清空**：当罚摸被压到 0 时，`state.penaltySource` 仍保留，后续 `applyPassTurn` 的清零逻辑依赖 `drawCardStack > 0` 才触发，导致 `penaltySource` 残留 | 高 | 在 `onActivate` 末尾加：`if (state.drawCardStack === 0) state.penaltySource = null;` |
| C-4 | **判定栈尚未实现**：用户规则要求"多个判定同时发生时按 LIFO 顺序结算"，当前判定同步执行，无真正的栈结构 | 低 | 未来在 `GameStateInternal` 中加 `judgmentStack: JudgmentEntry[]`，判定入栈后逐一结算 |
| C-5 | **亚双义变色后 `playability` 缓存可能不一致**（理论）：客户端 Lite 函数依赖 `statePatch` 携带的 `topCard`，技能改变手牌颜色后需要新的 `statePatch` 刷新，目前由广播机制保证——此问题实际不会发生，仅记录以供参考 | 低 | 现有广播机制已覆盖，无需修复 |

### 服务端层

| # | 问题 | 优先级 | 修复思路 |
|---|------|--------|---------|
| S-1 | **技能钩子在服务端重启后丢失**：`GameStateInternal` 从 SQLite 恢复时只恢复数据字段，`rules.hooks`（函数数组）无法序列化，重启后被动技能效果消失 | 高 | 在 `restoreRooms` 中，恢复 `in_game` 房间后读取 `game.characterAssignments`，重新调用 `applyCharacterSkills(game, assignments)` |
| S-2 | **`useSkill` 在 `snatch_window` 无额外约束**：部分技能（如御琴羽改变 `drawCardStack`）在抢牌窗口触发可能产生状态不一致 | 中 | 在 handler 的 `useSkill` 分支中，根据技能的 `allowedPhases` 字段（待设计）做阶段校验 |
| S-3 | **技能触发后不重置相位超时计时器**：御琴羽改变 `drawCardStack` 后，当前阶段的 `endsAt` 不变，玩家剩余决策时间不重置 | 低 | 根据需要在 `onActivate` 后调用 `setPhase` 刷新计时，但需明确是否真的需要重置 |

### 客户端层

| # | 问题 | 优先级 | 修复思路 |
|---|------|--------|---------|
| P-1 | **`pendingSkill` 与 `pendingWildCard` 可能同时非 null**：极端情况下两者共存会导致手牌点击行为歧义（当前代码 `pendingSkill` 优先，但逻辑不自然） | 中 | 进入技能选牌模式时强制 `setPendingWild(null, null, null)` |
| P-2 | **技能待命状态无超时/取消**：玩家进入"选目标"或"选牌"模式后不操作，`pendingSkill` 永远悬挂；对局超时自动流转后状态依然存在 | 中 | 在 `applyEvent` 的 `statePatch` / `gameStart` / `gameOver` 分支中清空 `pendingSkill` |
| P-3 | **成步堂目标高亮仅依据队伍，不检查手牌数**：客户端高亮所有敌方玩家，而非仅高亮"恰好 1 张牌"的敌人；点击不符合条件的目标后依赖 `actionRejected` 纠正 | 低 | 在 `isValidSkillTarget` 中额外检查 `gameState.players.find(...)?.handCount === 1` |
