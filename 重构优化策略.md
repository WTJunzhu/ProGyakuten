# 逆转 Uno 项目重构优化策略

## 一、项目现状

- **技术栈**：纯 TypeScript monorepo（npm workspaces），无 C++ 代码
- **规模**：22 个 TS 文件，约 4051 行代码
- **前端**：Vite + 原生 DOM 操作，main.ts 单文件 1330 行，无框架、无组件化、无状态管理
- **后端**：Node.js + ws 库，index.ts 单文件 838 行，房间管理和游戏逻辑混在一起
- **核心引擎**：packages/core，规则+动作+卡牌效果，钩子系统设计良好
- **协议层**：packages/protocol，ClientEvent（14种）+ ServerEvent（6种），类型完备
- **音频系统**：已有完整 audio.ts（450行），16 BGM + 7 SFX，支持情绪切换和循环点控制
- **联机方式**：cpolar 内网穿透 + WebSocket，无持久化、无认证、依赖房主在线
- **测试**：core 有 15 个测试，server 和 client 零测试

## 二、重构目标

不增加新功能，专注于提升现有项目的：
- **可维护性**：代码模块化，便于后续开发
- **可靠性**：测试覆盖，防止改动引入 bug
- **可用性**：脱离 cpolar，实现稳定可访问的联机
- **扩展性**：为未来角色系统、AI、多端联机打基础

## 三、重构范围总览

| 重构项 | 现状问题 | 目标状态 |
|--------|---------|---------|
| 测试体系 | 几乎空白（15个测试） | core 全覆盖 + server 单元测试 + 集成测试 |
| 前端架构 | 1330行单文件，命令式DOM | React + 组件化 + Zustand 状态管理 |
| 后端架构 | 838行单文件，无模块化 | 拆分为 connection/room/game-session/config |
| 代码重复 | 客户端重复实现出牌判定逻辑 | 统一到 core 包，客户端不再重复实现 |
| 持久化 | 纯内存，重启丢失 | SQLite 持久化 |
| 联机部署 | cpolar 隧道，依赖房主在线 | Serverless 平台（Railway/Render） |
| 构建部署 | 全手动 | GitHub Actions CI/CD |
| 认证系统 | 无，靠 playerId 字符串 | 匿名 ID 系统或房间密码 |
| 协议层 | 未预留扩展点 | 消息类型预留（角色/AI/聊天/观战） |

## 四、联机方式分析

### 当前方案优缺点

| 优点 | 缺点 |
|------|------|
| 权威服务器模型，防作弊 | 强依赖房主在线，关机即全掉线 |
| WebSocket 全双工，延迟低 | cpolar 隧道不稳定 |
| 重连机制完善（20秒宽限期） | 无持久化，重启丢状态 |
| 协议设计清晰，扩展性好 | 无认证，任何人可冒充 |
| 零运维成本 | 无 HTTPS/WSS，跨网络受限 |

### 目标方案：Serverless 平台

选择理由：不确定用户量，先用零成本方式试水。

**推荐组合**：
- 后端部署到 Railway 或 Render（免费额度够几十人用）
- 前端部署到 Vercel（免费 CDN）
- 总成本约 ¥50/年（域名）+ 可能的少量平台费用

**备选组合**：
- 阿里云轻量服务器 ¥60/月（完全可控，适合确定有用户量后切换）

## 五、重构阶段规划

### 阶段零：测试体系建设（1-2 周）

> 在动任何代码之前先建安全网。这是所有后续重构的前提。

**packages/core 测试补充：**
- 边界情况：空牌堆处理、手牌上限溢出、并发 seq 冲突
- Wild 组合各种边界：Wild + skip、Wild + reverse、Wild + +2 的合法性
- 罚摸连锁边界：+2→+2→+4→过牌的最终罚摸数
- Skip 约束：打出 skip 后下家的 allowedActions 是否正确
- 抢牌边界：Wild 组合抢牌、无牌可抢时的自动跳过

**apps/server 测试新增：**
- 房间管理：创建/加入/离开/房主转让
- 阶段超时：turn_main 超时自动摸牌、snatch_window 超时自动跳过
- 重连逻辑：断线宽限期、重连后状态恢复
- 消息处理：各种 ClientEvent 的合法/非法输入处理

**集成测试框架：**
- 模拟 2/4/6 人完整对局流程
- 创建房间→加入→开始→出牌→UNO→获胜→gameOver
- 验证每个阶段的 statePatch 广播内容

### 阶段一：前端架构重构（2-3 周）

**技术选型：**
- 框架：React + TypeScript
- 状态管理：Zustand（轻量，适合游戏状态）
- 样式：CSS Modules（替代 860 行内联样式）
- 构建：保留 Vite

**拆分方案：**

```
apps/client/src/
  components/
    TitleScreen.tsx      -- 标题画面
    Login.tsx            -- 登录/输入 playerId
    Lobby.tsx            -- 大厅（房间列表）
    Room.tsx             -- 房间等待（玩家列表、开始按钮）
    GameBoard.tsx        -- 游戏对局主界面
    Hand.tsx             -- 手牌区域
    CardView.tsx         -- 单张卡牌
    DiscardPile.tsx      -- 弃牌堆
    ColorPicker.tsx      -- Wild 选色弹窗
    AudioControls.tsx    -- 音量控制
    UnoButton.tsx        -- UNO 喊牌/检查按钮
  hooks/
    useGameSocket.ts     -- WebSocket 通信封装
    useGameState.ts      -- 游戏状态 hook
    useAudio.ts          -- 音频系统 hook
  stores/
    gameStore.ts         -- Zustand 游戏状态
    audioStore.ts        -- 音频状态
  utils/
    playability.ts       -- 出牌判定（从 core 导入，不再重复实现）
  App.tsx                -- 路由和视图切换
  main.ts                -- 入口
```

**关键改动：**
- 从 core 包导入出牌判定函数，消除 main.ts 中的重复实现
- audio.ts 接入 React useEffect 生命周期
- WebSocket 事件处理封装到 useGameSocket hook
- 保留现有的会话恢复机制（localStorage 中的 new_uno_session）

### 阶段二：后端架构重构 + 持久化（2 周）

**拆分方案：**

```
apps/server/src/
  connection.ts          -- WebSocket 连接管理、心跳、断线检测
  room.ts                -- 房间创建/加入/离开/状态查询
  game-session.ts        -- 游戏流程：回合推进、阶段超时、状态广播
  config.ts              -- 环境变量配置常量
  persistence.ts         -- SQLite 持久化层
  index.ts               -- 入口，组装各模块
```

**持久化方案：**
- 使用 better-sqlite3（Node.js SQLite 绑定，零外部依赖）
- 存储内容：房间状态、玩家信息、对局快照
- 写入时机：房间状态变更时写入，启动时从 SQLite 恢复
- 对 Serverless 平台友好：SQLite 文件可随实例存活，配合平台的持久卷使用

**广播优化：**
- 对比新旧 GamePublicState，只发送变化的字段
- 对 UNO 来说收益有限（数据量小），但代码更清晰

### 阶段三：消除代码重复 + 通信优化（1 周）

**消除重复：**
- 删除 main.ts 中的 `isCardPlayable()`、`isCardSnatchable()` 等函数
- 改为从 `@new-uno/core` 包导入
- 客户端 UI 高亮逻辑直接调用 core 的判定函数

**协议优化：**
- 增加消息版本号字段 `version: number`
- 为未来扩展预留空间（不实现，只预留类型位）

### 阶段四：Serverless 部署 + CI/CD（1 周）

**部署方案：**
- 后端：Railway 或 Render，连接 GitHub 仓库自动部署
- 前端：Vercel，自动部署
- 配置：环境变量管理（PORT、数据库路径等）

**CI/CD：**
- GitHub Actions：push 时自动跑测试
- 测试通过后自动部署到 staging 环境
- 手动触发部署到 production

**安全：**
- 配 SSL 证书（平台自动提供）
- WSS 协议（替代 ws）
- 前端 allowedHosts 配置

### 阶段五：认证 + 协议预留（1 周）

**认证方案（轻量）：**
- 方案A：房间密码（创建房间时设置，加入时验证）
- 方案B：匿名 ID 系统（首次访问生成唯一 ID，localStorage 持久化，跨设备用密码恢复）
- 推荐先做方案A，简单够用

**协议预留（不实现）：**
- `selectCharacter` 事件类型（为角色选角准备）
- `useSkill` 事件类型（为角色技能准备）
- `playerType: "human" | "ai"` 字段（为 AI 玩家准备）
- `chatMessage` 事件类型（为聊天准备）

## 六、保证原有逻辑的策略

### 三层防线

**第一层：行为快照测试（阶段零建立）**

测试"给定输入，应该得到什么输出"，不管内部实现。重构时只要行为不变，测试就通过。

关键测试场景：
- Wild 组合出牌的各种合法性判定
- 罚摸连锁的最终罚摸数计算
- Reverse 在罚摸链中的反弹效果
- Skip 约束下下家的可出牌范围
- UNO 检查的罚摸判定
- 抢牌的完全相同判定和 Wild 组合抢牌
- 手牌上限和补牌机制

**第二层：集成测试（端到端流程）**

模拟真实游戏流程，验证服务端+core 协同工作。抓模块之间的协作问题。

**第三层：手动对比回归（每个阶段结束）**

- 启动旧版本，跑一局完整对局，记录关键状态
- 启动新版本，执行相同操作
- 对比手牌显示、出牌高亮、音效触发、阶段切换、倒计时

### Bug 处理流程

1. **Git 小步提交**：每个逻辑单元改完就提交，方便定位引入 bug 的具体改动
2. **Git bisect**：测试挂了时用二分查找精确定位问题 commit
3. **按 Bug 类型处理**：
   - 规则判定错误 → 对照规则文档修正，补充测试用例
   - 状态不同步 → 对比新旧广播输出，检查字段遗漏
   - UI 交互异常 → 对比旧版事件处理流程
   - 协议字段缺失 → 对比新旧类型定义
   - 联机断线/重连 → 双标签页测试断线→重连流程
4. **回滚机制**：每个阶段完成后打 git tag，出问题可回滚到上一个 tag

### 每阶段检查清单

- [ ] 全部测试通过（`npm test`）
- [ ] 核心规则测试通过（`npm run -w @new-uno/core test`）
- [ ] 手动完成一局 2 人对局
- [ ] 手动完成一局 4 人对局
- [ ] 断线重连功能正常
- [ ] 音频系统正常播放
- [ ] 打 git tag
