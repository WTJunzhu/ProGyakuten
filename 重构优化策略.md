# 逆转 Uno 项目重构优化策略

## 一、项目现状

- **技术栈**：纯 TypeScript monorepo（npm workspaces），无 C++ 代码
- **规模**：22 个 TS 文件，约 4051 行代码
- **前端**：React 19 + Zustand 5 + Vite，组件化架构，已拆分为 12 个组件文件
- **后端**：Node.js + ws 库，已拆分为 8 个模块（index/connection/room/room-manager/handler/phase/broadcast/actions + auth/character/db/persistence）
- **核心引擎**：packages/core，规则+动作+卡牌效果，钩子系统设计良好
- **协议层**：packages/protocol，ClientEvent（18种）+ ServerEvent（10种），类型完备
- **音频系统**：audio.ts，BGM 场景切换 + SFX 音效，支持音量持久化
- **联机方式**：cpolar 内网穿透 + WebSocket，SQLite 持久化 + 账户认证
- **测试**：core 有完整单元测试，server 有持久化测试

## 二、重构目标

不增加新功能，专注于提升现有项目的：
- **可维护性**：代码模块化，便于后续开发
- **可靠性**：测试覆盖，防止改动引入 bug
- **可用性**：脱离 cpolar，实现稳定可访问的联机
- **扩展性**：为未来角色系统、AI、多端联机打基础

## 三、重构范围总览

| 重构项 | 现状问题 | 目标状态 |
|--------|---------|---------|
| 测试体系 | 几乎空白（15个测试） | core 全覆盖 + server 单元测试 + 集成测试 |
| 前端架构 | 1330行单文件，命令式DOM | React + 组件化 + Zustand 状态管理 |
| 后端架构 | 838行单文件，无模块化 | 拆分为 connection/room/game-session/config |
| 代码重复 | 客户端重复实现出牌判定逻辑 | 统一到 core 包，客户端不再重复实现 |
| 持久化 | 纯内存，重启丢失 | SQLite 持久化 |
| 联机部署 | cpolar 隧道，依赖房主在线 | Serverless 平台（Railway/Render） |
| 构建部署 | 全手动 | GitHub Actions CI/CD |
| 认证系统 | 无，靠 playerId 字符串 | 匿名 ID 系统或房间密码 |
| 协议层 | 未预留扩展点 | 消息类型预留（角色/AI/聊天/观战） |

## 四、联机方式分析

### 当前方案优缺点

| 优点 | 缺点 |
|------|------|
| 权威服务器模型，防作弊 | 强依赖房主在线，关机即全掉线 |
| WebSocket 全双工，延迟低 | cpolar 隧道不稳定 |
| 重连机制完善（20秒宽限期） | 无持久化，重启丢状态 |
| 协议设计清晰，扩展性好 | 无认证，任何人可冒充 |
| 零运维成本 | 无 HTTPS/WSS，跨网络受限 |

### 目标方案：Serverless 平台

选择理由：不确定用户量，先用零成本方式试水。

**推荐组合**：
- 后端部署到 Railway 或 Render（免费额度够几十人用）
- 前端部署到 Vercel（免费 CDN）
- 总成本约 ¥50/年（域名）+ 可能的少量平台费用

**备选组合**：
- 阿里云轻量服务器 ¥60/月（完全可控，适合确定有用户量后切换）

## 六、保证原有逻辑的策略

### 三层防线

**第一层：行为快照测试（阶段零建立）**

测试"给定输入，应该得到什么输出"，不管内部实现。重构时只要行为不变，测试就通过。

关键测试场景：
- Wild 组合出牌的各种合法性判定
- 罚摸连锁的最终罚摸数计算
- Reverse 在罚摸链中的反弹效果
- Skip 约束下下家的可出牌范围
- UNO 检查的罚摸判定
- 抢牌的完全相同判定和 Wild 组合抢牌
- 手牌上限和补牌机制

**第二层：集成测试（端到端流程）**

模拟真实游戏流程，验证服务端+core 协同工作。抓模块之间的协作问题。

**第三层：手动对比回归（每个阶段结束）**

- 启动旧版本，跑一局完整对局，记录关键状态
- 启动新版本，执行相同操作
- 对比手牌显示、出牌高亮、音效触发、阶段切换、倒计时

### Bug 处理流程

1. **Git 小步提交**：每个逻辑单元改完就提交，方便定位引入 bug 的具体改动
2. **Git bisect**：测试挂了时用二分查找精确定位问题 commit
3. **按 Bug 类型处理**：
   - 规则判定错误 → 对照规则文档修正，补充测试用例
   - 状态不同步 → 对比新旧广播输出，检查字段遗漏
   - UI 交互异常 → 对比旧版事件处理流程
   - 协议字段缺失 → 对比新旧类型定义
   - 联机断线/重连 → 双标签页测试断线→重连流程
4. **回滚机制**：每个阶段完成后打 git tag，出问题可回滚到上一个 tag

### 每阶段检查清单

- [ ] 全部测试通过（`npm test`）
- [ ] 核心规则测试通过（`npm run -w @new-uno/core test`）
- [ ] 手动完成一局 2 人对局
- [ ] 手动完成一局 4 人对局
- [ ] 断线重连功能正常
- [ ] 音频系统正常播放
- [ ] 打 git tag
